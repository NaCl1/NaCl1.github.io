2025软件系统安全赛 WEB JDBCParty

发表于2025-10-02|更新于2025-10-02
|阅读量:

JDBCParty

题目存在明显反序列化点

考点:

高版本 (jdk17) CVE-2022-39197
[[CVE-2022-39197 CS RCE]]

https://www.n4c1.top/2025/10/02/CVE-2022-39197-CS-RCE/

这个setURI就是最终的sink点, 题目存在 fastjson2和jackson依赖, 所以思路就是通过反序列化调用setter然后RCE

题目有oracleJDBC依赖 fastjson2 (2.0.37) 依赖 jackson依赖
这几个json库都是可以用来触发toString –> getter, 但我们最终要调用的其实是一个setter
思路是利用oracleJDBC依赖, 调用它的getter(getConnection)来打JNDI, 伪造恶意rmi服务器, 返回远程对象, 打高版本JDK的JNDI注入

首先是 readObect –> toString –> getter
前半段的 readObect –> toString, 我们在 CTF-Java-Gadget这个项目中就能找到几条可以用的, 这里选用EventListenerListReadObject2ToString
toString –> getter 直接选用 fastjson2原生反序列化链,

遇到的问题:

一

OK, 当我们直接写出这条链然后测试会发现, 确实触发到了getter, 但是在getConnection之前就触发了getCursorName(), 这个方法直接抛出错误

二

当我使用jackson来替换fastjson2时, 发现在序列化时就触发getConnection(), 导致生成不了poc

解决

由于题目存在spring依赖, 那么第一时间考虑的方法就是使用JdkDynamicAopProxy动态代理, 这个方法在 # 高版本Fastjson在Java原生反序列化 和 稳定jackson反序列化触发getter时都有用到
参考
# 关于java反序列化中jackson链子不稳定问题
getConnection的上层接口是RowSetInternal, 直接写出exp

这样生成的poc就可以完美触发到getConnection
之后就是伪造rmi server

由于是高版本jdk, 有module限制, rmiServer在编译和运行时分别需要设置命令行参数和jvm参数


之后按照[[CVE-2022-39197 CS RCE]]构造jar包和svg即可RCE
